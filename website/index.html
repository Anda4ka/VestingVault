How it works

From deploy to yield in 4 steps

VestingVault combines a battle-tested vesting mechanism with Synthetix-style reward distribution, all on Bitcoin L1.

01
⚙️

Deploy & Initialize

Deploy the WASM contract (no constructor args needed). Then call initialize(vestingToken, revenueToken) once as owner to configure token addresses. One-time, owner-only, replay-protected.

02
🔒

Lock Tokens

Owner adds a vesting schedule per beneficiary: amount, cliff (blocks), and linear duration. Tokens are pulled via transferFrom and locked in the vault.

03
💰

Deposit Revenue

Any address deposits revenue tokens (protocol fees, trading fees, etc.). The global rewardPerToken accumulator updates in O(1) — no iteration over beneficiaries.

04
⏳

Cliff Passes

Once the cliff block passes, tokens start vesting linearly. releasableAmount = vested − already released. Checked per-block with no rounding attacks.

05
🎯

Claim Everything

Beneficiary calls release() for vested tokens and claimRevenue() for their proportional share of all deposited revenue. State clears correctly.

Architecture

Built for correctness

Every design decision in VestingVault prioritizes safety, gas-efficiency, and mathematical correctness.

⏳

Block-Based Vesting

Cliff + linear release computed purely from block numbers — no timestamps, no oracle dependency. Deterministic and manipulation-resistant on Bitcoin L1.

Blockchain.block.numberU256
📊

Synthetix Accumulator

Revenue distributed via rewardPerToken global accumulator. Each deposit is O(1). Claims computed instantly from stored debt snapshots — no loops over beneficiaries.

rewardPerToken += (amount × 1e18) / totalLocked
🛡️

Persistent Reentrancy Guard

The lock flag is a StoredBoolean in blockchain storage — not an in-memory class field. Survives the per-call class re-instantiation unique to OPNet's execution model.

StoredBoolean(_locked, false)
🔢

SafeMath Everywhere

All 19 arithmetic operations use SafeMath.add/sub/mul/div. Zero floating point. Rounding always truncates toward zero, favoring the protocol — no slow drain possible.

SafeMath.mul(amount, PRECISION)
✅

CEI Pattern

State is always written before external Blockchain.call(). Checks → Effects → Interactions enforced in all 4 state-changing methods. Audited line-by-line.

updateReward() → state → callTransfer()
🔑

Correct Sender Check

Uses Blockchain.tx.sender (direct caller), never tx.origin. Prevents delegation attacks. Owner restricted to addVesting() only — revenue deposits are open.

Blockchain.tx.sender === _owner

Revenue distribution — verified

Example: Alice 70% locked, Bob 30% locked. Deposit 1 000 revenue tokens.

Deposit

rewardPerToken += 1000 × 10¹⁸ / 10000 = 10¹⁷

Claim (Alice — 7 000 locked)

earned = 7000 × 10¹⁷ / 10¹⁸ = 700 ✓
Bob earns 300. Total = 1 000. No dust left.

After partial release (Alice releases 3 000)

Next deposit 700 tokens:

How revenue flows

1. You call increaseAllowance(vault, amount)
2. Vault pulls tokens via transferFrom
3. rewardPerToken accumulates globally
4. Each beneficiary's pendingRevenue grows
5. No loops — O(1) per deposit ✓

⚙️ Step 1 — Initialize Vault (one-time)

Must be called once by owner right after deployment. Sets the vesting token and revenue token addresses. Cannot be changed after initialization.

Uses Vesting Token and Revenue Token from the config fields above.

Add Vesting Schedule

Owner only. Pulls tokens from your wallet via transferFrom and locks them for the beneficiary.

Vesting parameters

Cliff: blocks from now before anything vests
Duration: total blocks for 100% to vest
Linear: vested = amount × (block − start) / duration

Example: cliff=10, duration=100
→ After 10 blocks: 0 releasable
→ After 55 blocks: 55% releasable
→ After 100 blocks: 100% releasable
